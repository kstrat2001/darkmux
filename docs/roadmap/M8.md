# M8 — Capability Routing

> **Nav:** [← Roadmap](../../ROADMAP.md) · [All charters](./README.md) · [Open issues →](https://github.com/kstrat2001/darkmux/issues?q=is%3Aopen+label%3A%22theme%3Am8-routing%22)
>
> **Status:** sequenced (last theme; correctly decoupled from the earlier work) · **Target:** ships incrementally across the 1.x line (releases bundle lanes; no release owns a theme) · **Lead:** [#316](https://github.com/kstrat2001/darkmux/issues/316) (backend abstraction — must ship before #322)

This is the **charter** for milestone M8: the durable explanation of *what the theme is for, why, and where it's going*. The live, authoritative set of issues and their status lives on the [`theme:m8-routing` label view](https://github.com/kstrat2001/darkmux/issues?q=is%3Aopen+label%3A%22theme%3Am8-routing%22); this page is the reasoning behind it.

This is the endgame theme: pick the right local model for each piece of work automatically, by what it *can do*, not by a hardware-capacity label. It comes last on purpose. It composes on the runtime ([M5](./M5.md)), the fleet ([M6](./M6.md)), and the observability that measures whether a route was a good one ([M7](./M7.md)).

---

## The goal

Route each dispatch to the model that fits it, on a transparent capability match the operator can see and override. The model for the job is chosen by **capability**, not by the machine's RAM tier (that abstraction was tried and removed). The destination shape, already locked in the foundational refactors:

> The operator **lab-tests** their models → profiles **store** what each model is good at (a capability vector) → roles **request** capabilities (via their skills) → the runtime **matches** request to offering.

Two precursors had to land first, and did: the utility tier was decoupled from the profile (compactor became the standing `internal.utility` role, `ModelRole` removed, [#590](https://github.com/kstrat2001/darkmux/issues/590)), and the hardware-tier routing key was removed entirely (`DARKMUX_MACHINE_TIER` gone; hardware detection stays a lab concern, [#321](https://github.com/kstrat2001/darkmux/issues/321)). M8 builds the routing proper on that cleared ground: a backend abstraction, capability vectors, and continuous capability-based selection.

## Why this theme, why now

### What we measured (darkmux's own evidence)

- **The bake-off is capability routing done by hand.** darkmux already chooses models by capability: the [per-hardware-tier bake-off](https://github.com/kstrat2001/darkmux/issues/159) hires a 35B-A3B MoE for routine coding, holds a larger dense model as a heavy-reasoning reserve, keeps a coder-specialist for single-shot prose. That's a capability→role match an operator performs manually and re-derives whenever models turn over. M8 makes the match programmatic and the capability vector durable.
- **Tier was the wrong abstraction, and we proved it.** Machine-capacity tier was tried as a routing key and removed (#321/#590): a 35B reasoner and a 4B compactor on the *same* machine are different *capabilities*, not different *tiers*, and routing work by the machine's size conflated the two. The lesson (capability and role, not capacity) is exactly what this theme builds on.
- **Backend is a real routing dimension, not an afterthought.** The same model behaves differently across LMStudio vs Ollama vs llama.cpp (latency, throughput, feature support). If routing doesn't carry the backend as a first-class dimension from the start, it can't be retrofit. That's why the backend abstraction (#316) must ship *before* the router (#322).

### What the research says (verified citations)

Re-verified in the roadmap's [research pass](../../ROADMAP.md#how-we-decide); framing is *convergence, not priority*.

- **Routing a minority of queries to the expensive model preserves most of the quality.** RouteLLM hits 95% of the strong model's quality while routing only ~14% of queries to it ([arXiv 2406.18665](https://arxiv.org/abs/2406.18665)). → the economic case for capability routing: most work doesn't need the heavyweight, and a router can tell which does.
- **Skill Profiles map 1:1 onto darkmux's design.** Profile models by skill, match queries to profiles, expose the cost/performance trade-off transparently ([arXiv 2602.02386](https://arxiv.org/abs/2602.02386)). That's precisely the `capability → vector → match` shape, and precisely the transparency operator-sovereignty requires.
- **Routing is a recognized design space** with multiple paradigms to choose from deliberately ([arXiv 2603.04445](https://arxiv.org/abs/2603.04445)).

### Where the literature runs out — the white space M8 owns

Nearly all routing research assumes **cloud models on homogeneous hardware**. Capability routing across **local quantized models on a heterogeneous Apple-Silicon fleet** is genuinely under-served: the architecture is validated, but the setting is darkmux's to define. That's the differentiator the [roadmap leads with](../../ROADMAP.md), applied to the routing layer.

## Scope

The clusters below explain *why each issue is here*. For current status and the authoritative set, see the [`theme:m8-routing` issues](https://github.com/kstrat2001/darkmux/issues?q=is%3Aopen+label%3A%22theme%3Am8-routing%22).

### The routing build (sequenced — order matters here)

- **[#316](https://github.com/kstrat2001/darkmux/issues/316) — backend abstraction layer (`ModelBackend` trait) — lead, ship first.** Make the inference backend (LMStudio / Ollama / llama.cpp) a first-class abstraction so routing can carry it as a dimension. The one expensive-to-undo sequencing call: this *must* precede #322, because backend-as-routing-dimension can't be retrofit onto a router that didn't plan for it.
- **[#450](https://github.com/kstrat2001/darkmux/issues/450) — phase-1 capability abstraction.** The capability-vector groundwork (a model's `HashMap<Capability, f32>` offering, against a role's skill-derived request): the Skill-Profiles shape made concrete.
- **[#322](https://github.com/kstrat2001/darkmux/issues/322) — capability-based routing replaces tier labels (the endgame).** Continuous capability matching as the selection algorithm, on top of #316's backend dimension and #450's vectors. Selection resolves pin > score > default: the operator's explicit pin always wins.

### Recently resolved (the foundations this theme stands on)

- **Closed as shipped:** [#590](https://github.com/kstrat2001/darkmux/issues/590) (decouple the utility tier from the profile: compactor → `internal.utility` role, `ModelRole` removed, profile → capability wrapper) and [#321](https://github.com/kstrat2001/darkmux/issues/321) (remove `DARKMUX_MACHINE_TIER` entirely; hardware detection is the only local-tier source). Together these cleared tier-as-routing-key and established the role/capability split M8 routes over.

## Open scrub questions (for the holistic roadmap discussion)

- **Re-home [#70](https://github.com/kstrat2001/darkmux/issues/70)?** Its headline (compactor → standing utility model) shipped via #590/#602. The residual, predictive RAM headroom for not-yet-loaded models, is a *resource-allocation* concern, not capability *routing*. It likely belongs to the deferred parallel-work resource-allocation arc (KISS until real parallel dispatch arrives), and should probably move out of M8.
- **Crowdsourced capability auto-selection** is deliberately deferred: the locked destination has the operator lab-test their own models, not import a shared capability database. Worth confirming that stays out of M8 scope.
- **Where does pin-wiring on the internal-runtime path land?** Selection is `pin > score > default`; the explicit-per-role pin wiring on the internal path is still owed from the #590 arc and is a natural M8 dependency.

## Sequence

1. **#316** — backend abstraction (first, by necessity: the router can't add the backend dimension later)
2. **#450** — capability vectors (the request/offering shape; can proceed alongside #316)
3. **#322** — capability-based routing (the endgame, composing on both)

The wrong order is the one expensive mistake available in this theme: shipping #322 before #316 bakes a backend-blind router that then has to be torn up. The sequencing is the design decision.

---

## Success criteria

M8 is "done enough" when:

- [ ] The inference **backend is a first-class abstraction** (`ModelBackend`), so a model's identity includes where it runs (#316).
- [ ] Models carry a **capability vector** the operator establishes by lab-testing, and roles **request** capabilities via their skills (#450).
- [ ] A dispatch **routes to the matching model automatically**, by capability not tier, with the trade-off visible. An explicit operator **pin always overrides** the score (#322).
- [ ] The router is **backend-aware** from the first version, not retrofit.

Every routing decision stays **operator-sovereign**: the operator establishes the capability vectors (lab-tested, not imported), the router emits its reasoning + alternatives + confidence for audit, and a pin always wins over a score ([#44](https://github.com/kstrat2001/darkmux/issues/44)). The system proposes; the operator stays in the 20% of decisions that matter.
