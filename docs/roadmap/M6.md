# M6 — Fleet: Many Machines Become One

> **Nav:** [← Roadmap](../../ROADMAP.md) · [All charters](./README.md) · [Milestone on GitHub →](https://github.com/kstrat2001/darkmux/milestone/7)
>
> **Status:** sequenced (after [M5](./M5.md)) · **Target:** the event-sourcing core anchors around **v1.7.0** / spans **1.7.x**; the fleet-health cluster (#932) may land earlier as it addresses a live operator pain · **Lead item:** [#23](https://github.com/kstrat2001/darkmux/issues/23) (compat first), then [#280](https://github.com/kstrat2001/darkmux/issues/280) (the event-sourcing core)

This is the **charter** for milestone M6: the durable explanation of *what the theme is for, why, and where it's going*. The live, authoritative set of issues and their status lives on the [GitHub milestone](https://github.com/kstrat2001/darkmux/milestone/7); this page is the reasoning behind it.

M6 is the theme behind the project's tagline. [M4](./M4.md) and [M5](./M5.md) make a single machine's loop trustworthy; M6 makes a few of an operator's machines behave like **one** development environment.

---

## The goal

Let a single operator's couple of Macs over a tailnet function as one fleet: dispatch work to whichever machine should run it, manage mission state from any node, hand a workspace from one machine to another, and *understand the fleet's health and configuration* without guessing. The substrate already exists (Redis Streams over a tailnet: `darkmux:work` + `darkmux:flow`, `fleet status --deep`); M6 builds the coordination and coherence layers on top of it.

Two threads ladder to "many machines become one":

1. **Coordination** — version-compatible cross-machine dispatch, event-sourced mission state manageable from any node, and workspace handoff between machines. Make the fleet *act* as one.
2. **Coherence** — a fleet-aware `doctor` that diagnoses machine-level and fleet-level config drift and generates the remediation, plus the config surface to make a fleet position declarable. Make the fleet *legible and configurable* as one.

The design target is deliberately **heterogeneous and single-operator**: a high-memory laptop as the inference peer, a smaller always-on machine as the hub, joined over a tailnet the operator controls. The trust boundary is the tailnet, not enforcement in darkmux's code (see [`DESIGN.md` → What darkmux is NOT](../../DESIGN.md#what-darkmux-is-not)).

## Why this theme, why now

### What we measured (darkmux's own evidence)

- **The config surface got too complex; it confused its own author.** The operator, an engineer who built darkmux, got tangled configuring his own two-machine fleet. That's a failure of the operator-sovereignty principle ("the operator never has to wonder where a decision came from"), and it's *the* reason the fleet-health / config-coherence system (#932) exists. The fix is structural: a fleet-aware `doctor` generated from live state, not more static docs that drift per-machine.
- **A machine dropping off shouldn't be a mystery.** Real fleet operation surfaced the failure mode where one machine silently falls off the substrate and the operator can't tell *why* (route down? daemon down? auth? stale password?) without manual probing. Fleet-level coherence checks turn that into a diagnosis.
- **Heterogeneity is the rule, not the exception.** The reference fleet is genuinely uneven: a 128GB laptop and a 32GB always-on machine play different roles (inference peer vs hub). Tier-as-routing-key was tried and removed (#590); the lesson is that heterogeneity belongs in *capability* and *role*, not a capacity label.

### What the research says (verified citations)

Re-verified in the roadmap's [research pass](../../ROADMAP.md#how-we-decide); framing is *convergence, not priority*.

- **Event-sourcing is the canonical distributed-agent backbone:** immutable events as the source of truth give replay, audit, and coordination for free ([akka.io](https://akka.io/blog/event-sourcing-the-backbone-of-agentic-ai)). → validates the `darkmux:flow` substrate and the event-sourced mission state (#280/#282) that M6 builds on.
- **Capability discovery via semantic matching**, not name lookup, is the standard for inter-agent systems ([arXiv 2511.19113](https://arxiv.org/abs/2511.19113)). → grounds routing by what a machine *can do* (the M8 successor to the removed tier label) and the heterogeneous-fleet model.
- **State commit / revert / branch primitives** for agent work map cleanly onto git ([AgentGit, arXiv 2511.00628](https://arxiv.org/abs/2511.00628)). → grounds the workspace-handoff-via-git-refs design (#755).

### Where the literature runs out — the white space M6 owns

Nearly all distributed-agent research assumes **cloud or homogeneous hardware**. darkmux's premise (a heterogeneous local fleet of Apple-Silicon Macs over a tailnet, running quantized models) is genuinely under-served. That gap is the differentiator the [roadmap leads with](../../ROADMAP.md): the event-sourced two-tier local fleet is darkmux's to define, not to follow.

## Scope

The clusters below explain *why each issue is here*. For current status and the authoritative set, see the [milestone](https://github.com/kstrat2001/darkmux/milestone/7).

### Coordination core

- **[#23](https://github.com/kstrat2001/darkmux/issues/23) — cross-machine version-mismatch hardening (lead / Phase 1).** Compat must precede coordination: machines that disagree on the wire format can't safely share a substrate. The gate before everything else here.
- **[#280](https://github.com/kstrat2001/darkmux/issues/280) — event-sourced mission state across the fleet.** Manage a mission from any node. The core that unblocks priority/pause and workspace handoff; grounded in the event-sourcing research.
- **[#282](https://github.com/kstrat2001/darkmux/issues/282) — mission priority + pause/resume from any node.** Composes on #280.
- **[#755](https://github.com/kstrat2001/darkmux/issues/755) — cross-machine workspace handoff via git refs + worktrees.** "The missing half of many-machines-become-one": hand the actual working tree between machines using git as the transport (AgentGit-grounded), superseding the earlier iCloud-fallback idea.
- **[#852](https://github.com/kstrat2001/darkmux/issues/852) — cross-machine causality tracing.** Parent/child spans across the fleet boundary, so a dispatch routed to another machine stays legible in the observability stream.

### Fleet health + config coherence (the #932 system)

The structural answer to the config-complexity pain: `doctor` grown from one machine to the whole fleet, generated from live state rather than written as drift-prone docs.

- **[#932](https://github.com/kstrat2001/darkmux/issues/932) — EPIC: network-health + config-coherence system (fleet doctor).**
- **[#934](https://github.com/kstrat2001/darkmux/issues/934) — doctor Layer 1: machine-coherence rules + verdict banner.** Is *this* machine internally consistent?
- **[#935](https://github.com/kstrat2001/darkmux/issues/935) — doctor `--fleet` (Layer 2): fleet coherence matrix + two-hub split-brain detection.** Is the *fleet* consistent across machines?
- **[#933](https://github.com/kstrat2001/darkmux/issues/933) — `fleet.mode` config field (hub | peer | standalone).** Let a machine *declare* its fleet position so coherence checks have an intent to check against.
- **[#936](https://github.com/kstrat2001/darkmux/issues/936) — turnkey hub:** the daemon supervises its own Redis when `fleet.mode=hub`, closing the "stand up the coordination substrate" friction.
- **[#937](https://github.com/kstrat2001/darkmux/issues/937) — `darkmux config set/get` verb** for pasteable config remediation (so doctor's findings come with a one-command fix).

### Model lifecycle + heterogeneous hardware

- **[#535](https://github.com/kstrat2001/darkmux/issues/535) — unified model load/swap lifecycle:** reconcile + leases + truth-stamping, balancing lab reproducibility against crew warmth.
- **[#597](https://github.com/kstrat2001/darkmux/issues/597) — multi-endpoint-per-machine:** per-model inference-endpoint addressing (an eGPU/CUDA tier, N LMStudio ports), the concrete heterogeneous-hardware case.
- **[#117](https://github.com/kstrat2001/darkmux/issues/117) — consumer / micro / mobile profiles** for low-RAM machines (re-scoped to the real need, dropping the issue's obsolete role-family and tier-routing framing). Touches M8 capability routing.

## Sequence

1. **#23** — version-compat (the gate; coordination can't precede it)
2. **#280** — event-sourced mission state (the core that unblocks #282 + #755)
3. **#282 / #755 / #852** — priority/pause, workspace handoff, causality tracing on top of the core
4. **#932 cluster** — the fleet-health system runs partly in parallel: it's substrate-light (it reads live state + config) and addresses a live pain, so #934 (machine-coherence) + #933 (`fleet.mode`) can land early, with #935 (`--fleet`) following once there's declared intent to check

## Success criteria

M6 is "done enough" when:

- [ ] Machines on different darkmux versions **fail safe**, not silently corrupt the shared substrate (#23).
- [ ] A mission's state is **event-sourced and manageable from any node**: propose on one machine, pause from another (#280, #282).
- [ ] A workspace **hands off between machines** via git refs without manual rsync/iCloud (#755).
- [ ] `darkmux doctor --fleet` **diagnoses fleet-level drift** (split-brain, version skew, config incoherence) and **generates the remediation**, so the operator never has to reverse-engineer why a machine fell off (#932 cluster).
- [ ] A machine can **declare its fleet position** (`fleet.mode`) and stand up its own coordination substrate turnkey (#933, #936).

Every cross-machine action stays **operator-sovereign**: provenance is operator-asserted (no authenticated identity inside the trust boundary), config is declared not inferred, and the fleet doctor *surfaces and suggests* remediation rather than auto-applying it ([#44](https://github.com/kstrat2001/darkmux/issues/44)).

## Target release

Per the roadmap principle, [milestones are themes, not version numbers](../../ROADMAP.md).

- The **event-sourcing core** (#23 → #280) anchors around **v1.7.0** and spans **1.7.x**.
- The **fleet-health cluster** (#932) may land earlier: it's substrate-light and addresses a live config-coherence pain, so it isn't gated behind the coordination core.
- Compat hardening (#23) lands first within the theme by necessity.
