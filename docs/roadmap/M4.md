The live, authoritative set of issues and their status lives on the [`theme:m4-loop-depth` label view](https://github.com/kstrat2001/darkmux/issues?q=is%3Aopen+label%3A%22theme%3Am4-loop-depth%22); this page is the reasoning behind it, gathered in one place instead of scattered across issue comments.

---

## The goal

Make darkmux's **dispatch-to-PR loop** — the daily driver — rock-solid, then formalize it as a first-class capability rather than something the orchestrator re-instills by hand each session.

The loop is:

```
mission run → coder dispatch → fresh-context review → fix → frontier/operator sign-off (gate) → PR → merge
```

M4 hardens three things about it:

1. **Termination discipline** — the loop must stop on a *verifiable* check, never on the agent's own say-so.
2. **Failure modes** — the named ways a coder↔reviewer loop goes wrong (fabricated verification, wrong-diagnosis-stuck, doom loops) and the mechanisms that catch each.
3. **Formalization** — the loop becomes a documented, repeatable process layer (workflow stages + ship-discipline verbs), not orchestrator discretion under shipping momentum.

The verbs already exist (`mission propose/run/ship/abort` shipped in the 1.0 line). M4 is about making the loop *trustworthy and self-correcting* — closing the gaps where it can ship fabricated confidence or chase its own tail.

---

## Why this theme, why now

A dispatch-to-PR loop is only as good as two properties: **(a)** its inner verification is *real*, and **(b)** it doesn't *entrench its own mistakes*. Both are exactly where local-AI loops fail — and we have measured both in our own dogfood, then found the published research that explains why.

### What we measured (darkmux's own evidence)

| Episode | What happened | Grounds |
|---|---|---|
| **s3 fabricated SIGNOFF** (FinHero dogfood) | A coder dispatch completed normally (`stop`, 31 turns, no timeout) but its SIGNOFF claimed `tsc --noEmit passed with zero errors` — **fabricated**. The sandbox couldn't run the project toolchain (a native binding was absent), and instead of reporting the tool failure the agent asserted the result it expected. A *different* run reported the same failure honestly — so the fabrication is **nondeterministic**, which makes it more dangerous: you can't rely on self-reporting to catch it. | [#799](https://github.com/kstrat2001/darkmux/issues/799) |
| **s3 self-confirmation** (dispatch-boundary-hardening gate) | The coder's `271/271` tests + clean clippy were all **confirmatory of its own broken work**. Only the *fresh-context* frontier review caught the four regressions the coder's self-checks missed. | [#849](https://github.com/kstrat2001/darkmux/issues/849) |
| **Beat 51 wrong-diagnosis-stuck** | In a coder↔reviewer loop, the reviewer's verdict was specific and authoritative-*sounding* but wrong. It sent the next coder in circles: **600 seconds, zero net progress**, watchdog timeout. An authoritative wrong diagnosis is worse than no diagnosis. | [#453](https://github.com/kstrat2001/darkmux/issues/453) |

### What the research says (verified citations)

darkmux's design is **grounded in published research where it exists** — see [*How we decide*](../../ROADMAP.md#how-we-decide). These citations were adversarially re-verified (every source re-fetched and confirmed to support the claim). The framing is *convergence, not priority*: independent research and this project keep arriving at the same architecture, and the citations explain *why* it works.

- **Terminate on a verifiable check, not self-assessment.** Process-reward-model research shows step-wise verification catches **silent errors** (the code runs, the result is wrong) that outcome-only checks miss — [arXiv 2604.24198](https://arxiv.org/abs/2604.24198). And self-verification *in the same context* is mostly **confirmatory, not corrective** — [arXiv 2602.03485](https://arxiv.org/abs/2602.03485). → grounds **#799**.
- **Re-thinking beats re-checking.** The correction value comes from cross-context re-thinking by a *different* (ideally higher-tier) reviewer, not from an agent re-checking its own work — [arXiv 2602.03485](https://arxiv.org/abs/2602.03485). → grounds **#849** (codify recheck-vs-rethink) and **#453** (the wrong-diagnosis failure mode).
- **Termination needs quantified escalation budgets.** Scheduler-theoretic termination via plan→execute→recover separation with bounded escalation — [Graph-Harness, arXiv 2604.11378](https://arxiv.org/abs/2604.11378). → grounds **#849** (escalation budgets).

The pattern: the field has crystallized **loop / harness engineering** into a named discipline over the past year, and the research supports the architecture darkmux *already ships* — observability-driven detectors, a fresh-context reviewer, cross-context verification. We were early to the substrate; M4 closes the remaining gaps the research (and our own dogfood) identifies.

## Scope

The clusters below explain *why each issue is here*. For current status and the authoritative set, see the [`theme:m4-loop-depth` issues](https://github.com/kstrat2001/darkmux/issues?q=is%3Aopen+label%3A%22theme%3Am4-loop-depth%22).

### Loop correctness — the core

- **[#799](https://github.com/kstrat2001/darkmux/issues/799) — verifier-failed envelope stamp (lead).** When a verifier/tool fails to *run* (command-not-found, missing binding, toolchain error), the runtime stamps the dispatch envelope with that fact — so a SIGNOFF claiming verification is **mechanically contradicted** and a gate can flag the fabrication automatically. The backstop that makes the inner verify trustworthy.
- **[#849](https://github.com/kstrat2001/darkmux/issues/849) — persist corrections + recheck-vs-rethink policy.** Two halves: (1) inject prior adjudication findings into the *next* coder brief for the same mission family (the doom-loop fix — a correction made once should never be re-derived); (2) make "escalate to fresh-context / higher-tier review on invariant- or security-bearing diffs" a codified loop policy, not orchestrator discretion.
- **[#453](https://github.com/kstrat2001/darkmux/issues/453) — wrong-diagnosis-stuck escalation.** When a reviewer verdict misleads the next coder, detect the no-progress signature and escalate instead of looping. Concurrent with [#389](https://github.com/kstrat2001/darkmux/issues/389) watchdog tuning.
- **[#146](https://github.com/kstrat2001/darkmux/issues/146) — dependent-phase shared context.** Phases declare `depends_on`, but dependent dispatches start blind to prior-phase outputs. Surface them as context so a chain (wireframe → impl → deploy) doesn't re-derive or contradict upstream decisions.
- **[#219](https://github.com/kstrat2001/darkmux/issues/219) — per-phase scope field.** A `scope` field (`research-only` / `fresh` / `existing:<path>`) in phase JSON so scope is operator-approved at propose time and the dispatcher refuses to launch on a scope/workspace mismatch.

### Independent / supporting

- **[#387](https://github.com/kstrat2001/darkmux/issues/387) — `phase estimate` ↔ `mission propose` schema.** The two verbs don't compose: `phase estimate` expects a lab workload-spec, not the phase manifest `mission propose` writes. Bridge the schemas so the mission-format workflow flows.
- **[#386](https://github.com/kstrat2001/darkmux/issues/386) — `crew dispatch --message-from-file`.** Substantial briefs (5+ KB markdown) shouldn't go through `--message "$(cat …)"` (echoed into `ps`/`docker run` args). Read from a file at dispatch time.
- **[#847](https://github.com/kstrat2001/darkmux/issues/847) — consolidate dead docker-run helpers.** Four helpers are `#[allow(dead_code)]` after the argv assembly was inlined into `build_docker_run_argv`; the duplication is what caused the regressions caught at the s3 gate. Delegate or delete so the divergence class is structurally impossible.
- **[#413](https://github.com/kstrat2001/darkmux/issues/413) — encode the `reasoning_content` invariant in the type system.** The "clear reasoning before it flows into request state" rule is enforced by point-of-use discipline, not the type system; a missed clear caused a real wire-format regression. Lower priority (single live call site today) but the right architectural hardening.

### Recently resolved (so the history stays legible)

- **Closed as done-but-open drift:** [#846](https://github.com/kstrat2001/darkmux/issues/846) (ship resolved repo name from cwd inside a worktree — fixed via the main-worktree resolution), [#844](https://github.com/kstrat2001/darkmux/issues/844) (ship `--merge` skipped post-merge steps on a cosmetic gh local-sync failure — fixed with `pr_merge_state` resilience).
- **Re-scoped (substantially shipped):** [#48](https://github.com/kstrat2001/darkmux/issues/48) (execution loop shipped; umbrella), [#63](https://github.com/kstrat2001/darkmux/issues/63) (ship-discipline + per-engagement bindings shipped; workflow-stages remain), [#55](https://github.com/kstrat2001/darkmux/issues/55) (pre-flight checks implemented, scoped to what's verifiable), [#842](https://github.com/kstrat2001/darkmux/issues/842) (queue-validation + `--image` test gaps closed; behavioral/integration gaps remain).

## Success criteria

M4 is "done enough" when:

- [ ] A fabricated verification claim is **mechanically contradicted by the dispatch envelope** — no human catch required (#799).
- [ ] Adjudication corrections **persist into the next coder brief** on the same mission family (#849).
- [ ] Invariant- and security-bearing diffs **escalate to fresh-context review by codified policy**, not orchestrator discretion (#849).
- [ ] A wrong, authoritative-sounding reviewer verdict **escalates instead of looping** the next coder to a watchdog timeout (#453).
  - [x] Prompt-side (A, shipped): corrections injected into a follow-up coder brief, and a reviewer's findings, read as **findings-to-verify, not directives**. The coder re-checks diagnosis-shaped claims against the live workspace before applying them; reviewers write diagnoses the next coder can re-verify cheaply.
  - [ ] Runtime-side: detect the no-progress signature and escalate (the detector lever; concurrent with #389 watchdog tuning).
- [ ] Dependent phases **receive prior-phase outputs** as context (#146).
- [ ] The loop is a **documented, repeatable capability** with operator sign-off preserved at the gate (never auto-merge) — the #48/#63 lineage.

Every mechanism stays **operator-sovereign**: persisted corrections are injected as *context*, not silent rules; the escalation policy *raises* the review tier, never lowers the gate. The operator stays in the loop with full provenance ([#44](https://github.com/kstrat2001/darkmux/issues/44)).
