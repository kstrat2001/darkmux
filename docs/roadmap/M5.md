# M5 — Runtime / Agent-Loop Robustness

> **Nav:** [← Roadmap](../../ROADMAP.md) · [All charters](./README.md) · [Milestone on GitHub →](https://github.com/kstrat2001/darkmux/milestone/6)
>
> **Status:** next (sequenced after [M4](./M4.md), overlapping its tail) · **Target:** groundwork rides the **1.5.x** line, the theme anchors around **v1.6.0** · **Lead item:** [#422](https://github.com/kstrat2001/darkmux/issues/422)

This is the **charter** for milestone M5 — the durable explanation of *what the theme is for, why, and where it's going*. The live, authoritative set of issues and their status lives on the [GitHub milestone](https://github.com/kstrat2001/darkmux/milestone/6); this page is the reasoning behind it.

Where [M4](./M4.md) hardens the **outer** loop (coder → review → gate → PR), M5 hardens the **inner** loop M4 runs on: the single dispatch — one model, in one container, working a turn at a time until it stops, struggles, or escalates.

---

## The goal

Make a single dispatch **fail well**. A local model in an agent loop fails in characteristic ways — re-reading the same file, re-reasoning the same dead end, hammering a tool that keeps erroring, emitting reasoning until it hits the token cap with nothing to show. darkmux already *detects* these (the cycle / reasoning-loop / failure-cascade / cadence-drift detectors ship today, each writing a trajectory event). M5 is about **acting** on what they see, correctly and legibly:

1. **Classify failures** instead of swallowing them. A dispatch error today is an opaque string; it can't be routed. M5 gives the runtime a structured error taxonomy (transient / permanent / operator-actionable) so retry-vs-escalate-vs-surface becomes a decision, not a guess.
2. **Recover, don't discard.** Salvage well-formed work from a capped turn, promote plain-text tool calls back to structured, retry a transient failure within a bounded budget — each as a visible trajectory event, not a silent drop.
3. **Tune the biggest lever with data.** Compaction has the largest measured wall-clock impact, so its defaults get earned in the lab, not asserted — which first requires lab workloads that actually trip it.

The unifying principle is **observability before intervention** (already doctrine — see [`DESIGN.md`](../../DESIGN.md#runtime-resilience-struggle-detection--feedback-injection)): every detector is visible in the trajectory, every nudge is attributable to a named signal, every bound is operator-tunable, and nothing silently changes a dispatch without leaving a record of why. M5 builds the recovery layer *on top of* that observability — never around it.

## Why this theme, why now

The detectors gave darkmux **eyes** on inner-loop struggle. The gap M5 closes is that, today, what the runtime mostly does with what it sees is *write it down and keep going* — the MVP was visible struggle, not recovery. Two things make recovery the right next investment: the published research says the inner loop is exactly where long-horizon agents break, and darkmux's own dogfood keeps surfacing the failure modes the research predicts.

### What we measured (darkmux's own evidence)

- **Reliability decays over a long horizon.** darkmux's longest self-building dispatch ran 106 turns; lab runs of the long-agentic workload routinely stretch into the tens of turns. The further a dispatch runs, the more it drifts toward repetition — which is *why* the detectors exist, and why acting on them early matters more the longer the loop.
- **Opaque errors can't be routed.** When a dispatch fails today, the result is an `anyhow` string. There's no machine-readable answer to "is this transient (retry), permanent (bail), or operator-actionable (surface and stop)?" — so the loop can't make the recovery decision the detectors' signals would otherwise inform.
- **The biggest lever has a measurement gap.** Compaction is the harness knob with the largest measured wall-clock impact, but a key lab workload (medium-coding) reliably *never trips* the default threshold — so there's no before/after baseline to grade the structured-slot compactor against the narrative one. You can't tune what you can't make fire.

### What the research says (verified citations)

These were adversarially re-verified in the roadmap's [research pass](../../ROADMAP.md#how-we-decide) (every source re-fetched, confirmed to support the claim). Framing is *convergence, not priority*: the field arrived at the same architecture darkmux was already building.

- **Self-healing = detect + classify + replan.** Robust agent loops pair failure *detection* with an error *taxonomy* and adaptive *recovery* — the taxonomy is the hinge between seeing a failure and acting on it ([arXiv 2605.06737](https://arxiv.org/abs/2605.06737)). → grounds the lead item, **#422** (structured error classification), and the recovery epic **#417**.
- **Failure-aware observability maps failure modes to *online* trace signals.** The signals that catch agent failure are available *during* the run, not just in post-hoc logs — which is exactly what edge-triggered detectors over a live trajectory provide ([arXiv 2606.01365](https://arxiv.org/abs/2606.01365)). → validates the detector-driven design and the recovery layer M5 builds on it.
- **Reliability decays super-linearly over long horizons**, and sliding-window signals can catch the coherent→looping collapse before it bottoms out ([arXiv 2603.29231](https://arxiv.org/abs/2603.29231)). → grounds tightening the loop detectors (**#476**) and the salvage/escalation budgets (**#481**, **#467**).
- **Tool preferences are systematically unreliable** — positional bias and description sensitivity mean a model's tool-call output can't be taken at face value, and instructions don't reliably override the learned bias ([arXiv 2505.18135](https://arxiv.org/abs/2505.18135)). → validates defensive parsing of model-emitted tool calls (**#411** narrowing the plain-text promoter heuristic).

The pattern matches [M4](./M4.md) and the [compaction design](../../DESIGN.md#compaction-tiers-structured-slots-and-graceful-degradation): darkmux shipped the substrate (detectors, feedback injection, salvage) before the literature named it; M5 closes the recovery layer the research says completes it.

## Scope

The clusters below explain *why each issue is here*. For current status and the authoritative set, see the [milestone](https://github.com/kstrat2001/darkmux/milestone/6).

### Error taxonomy + recovery — the core

- **[#422](https://github.com/kstrat2001/darkmux/issues/422) — structured error classification (lead).** Replace opaque `anyhow` strings with a typed taxonomy (transient / permanent / operator-actionable). This is the prerequisite the research names: detection plus *taxonomy* is what makes recovery a decision instead of a guess. Everything downstream (retry policy, escalation routing, what gets surfaced to the operator) composes against it.
- **[#417](https://github.com/kstrat2001/darkmux/issues/417) — EPIC: automate the human-in-the-loop recovery behaviors.** The umbrella for turning detector signals into bounded automatic recovery (retry-with-backoff, escalation routing) — the behaviors openclaw's conversation-mode design doesn't need but darkmux's autonomous dispatch does.
- **[#476](https://github.com/kstrat2001/darkmux/issues/476) — tighten reasoning-loop re-arm semantics.** Permissive single-divergence clear vs strict multi-distinct-hash clear — make the loop detector harder to fool into thinking the model escaped a loop it's still in.
- **[#481](https://github.com/kstrat2001/darkmux/issues/481) — per-turn-cap salvage budget.** Cap how many times one dispatch may salvage a capped turn (`MAX_SALVAGES`) before escalating, with a named `EscalationReason` — so salvage is a bounded recovery, not an infinite one.
- **[#467](https://github.com/kstrat2001/darkmux/issues/467) — infraction cool-down / strike system (deferred).** Escalating termination on sustained pathology signals. Filed deferred; the heaviest-handed recovery, gated behind the taxonomy + budgets landing first.

### Lab-driven compaction tuning

- **[#366](https://github.com/kstrat2001/darkmux/issues/366) — workload sizing so compaction trips.** The medium-coding workload never reaches the default threshold, so there's no baseline to grade tier-2 against the narrative compactor. Fix the lab before tuning the runtime.
- **[#355](https://github.com/kstrat2001/darkmux/issues/355) — lab-driven v0.1 parameter tuning.** The structured-slot schema defaults are *reasonable stakes from research*, measured-against, not declared-correct. This is the empirical validation pass that refines them. The lab *is* the tune-up system.
- **[#400](https://github.com/kstrat2001/darkmux/issues/400) — re-baseline the long-agentic test + tune the verification rule.** Re-run the canonical long-agentic workload under the current (no-in-container-verification) coder rule and re-establish the published baselines, then tune the capability-aware verification rule. The seed of the differentiator the roadmap leads with — [the harness that learns from its own failure record](https://github.com/kstrat2001/darkmux/issues/400).

### Streaming / promoter follow-ups

- **[#411](https://github.com/kstrat2001/darkmux/issues/411) — narrow the plain-text tool-call heuristic to objects/arrays.** The promoter currently treats any JSON-ish prefix (including bare scalars) as parseable; narrow it so a tool argument that merely *looks* like a literal isn't silently retyped. Grounded in the tool-unreliability research above.
- **[#412](https://github.com/kstrat2001/darkmux/issues/412) — emit `model.reasoning` on non-streaming responses with separate-field reasoning.** Observability completeness — the inline-think-tag and streaming paths emit it; close the non-streaming separate-field gap.
- **[#410](https://github.com/kstrat2001/darkmux/issues/410) — streaming-path integration test for XML-in-reasoning recovery.** Lock the [#406](https://github.com/kstrat2001/darkmux/issues/406) promoter behavior on the streaming path with an integration test.
- **[#243](https://github.com/kstrat2001/darkmux/issues/243) — test-coverage gaps:** chat-streaming, mission-compile, plan-migration.

### Runtime infra / consistency

- **[#199](https://github.com/kstrat2001/darkmux/issues/199) — container lifecycle:** per-engagement vs daemon vs persistent caches — how long a runtime container lives and what it reuses.
- **[#404](https://github.com/kstrat2001/darkmux/issues/404) — `lab run --runs N` auto-restores canonical sandbox state** between dispatches for external workloads (composes with the lab-tuning cluster — repeatable runs need a clean start each time).
- **[#244](https://github.com/kstrat2001/darkmux/issues/244) — post-[#220](https://github.com/kstrat2001/darkmux/issues/220) consistency drive-bys:** mission_id plumbing, dir cleanup, reasoning emit-sites, exit codes, naming.
- **[#353](https://github.com/kstrat2001/darkmux/issues/353) — compaction-redesign docs tail.** The [`DESIGN.md`](../../DESIGN.md) piece landed; remaining = surface the `compaction.strategy` override flag in README, refresh the lab guide, re-measure the long-agentic workload's expected block (overlaps the tuning cluster).

### Recently resolved (so the history stays legible)

- **Closed as shipped:** [#354](https://github.com/kstrat2001/darkmux/issues/354) (structured-state schema — `StructuredCompactionOutput` shipped), [#205](https://github.com/kstrat2001/darkmux/issues/205) (streaming chat-completions — `chat_streaming` + `ChunkAccumulator` + `model.partial`), [#854](https://github.com/kstrat2001/darkmux/issues/854) (stale `usage.prompt_tokens` — the frozen-turns / `effective_prompt_tokens` fix).
- **Closed as resolved-by-redesign:** [#389](https://github.com/kstrat2001/darkmux/issues/389) (the fixed 600s watchdog was replaced by the inactivity timeout [#458](https://github.com/kstrat2001/darkmux/issues/458), which resets on proof-of-work and solves the post-compaction-retrace case directly) and [#347](https://github.com/kstrat2001/darkmux/issues/347) (a hard wall-clock cap is superseded by that same inactivity model — the runtime is deliberately inactivity-based, not wall-clock-based).

## Sequence

**[#422](https://github.com/kstrat2001/darkmux/issues/422) leads** — the research names the error taxonomy as the hinge: detection without classification can't drive recovery. Then:

1. **#422** — structured error taxonomy (the prerequisite everything composes against)
2. **#417** — recovery behaviors on top of the taxonomy (retry/backoff, escalation routing); **#476** + **#481** tighten the loop detectors and bound salvage concurrently
3. **#366 → #355 / #400** — fix the lab so compaction trips, then tune defaults against measurement (the lab-tuning chain; #404 supports repeatable runs)
4. **#411 / #412 / #410 / #243** — promoter + streaming follow-ups, folded in as they fit
5. **#467** — the strike system last, gated behind the taxonomy + budgets

M4 and M5 aren't cleanly sequential — they overlap. M5's error-taxonomy groundwork can begin while M4's loop-correctness work wraps, and M4's escalation policy ([#849](https://github.com/kstrat2001/darkmux/issues/849)) pairs naturally with M5's recovery behaviors.

## Success criteria

M5 is "done enough" when:

- [ ] A dispatch failure carries a **structured classification** (transient / permanent / operator-actionable), not an opaque string (#422).
- [ ] The runtime **acts on detector signals** with bounded automatic recovery — retry transient failures, escalate sustained pathology — each as a visible trajectory event (#417).
- [ ] Salvage and loop-recovery are **budget-bounded** with named escalation reasons, never infinite (#481, #476).
- [ ] Compaction defaults are **graded in the lab** against a workload that actually trips them, with re-established long-agentic baselines (#366, #355, #400).
- [ ] Model-emitted tool calls are **parsed defensively** — a value that merely looks like a literal isn't silently retyped (#411).

Every recovery stays **operator-sovereign**: every detector observable in the trajectory, every nudge attributable to a named signal, every bound operator-tunable, nothing silently changed without a record of why ([#44](https://github.com/kstrat2001/darkmux/issues/44)).
