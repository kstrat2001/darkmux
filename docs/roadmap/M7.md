This is the **charter** for milestone M7: the durable explanation of *what the theme is for, why, and where it's going*. The live, authoritative set of issues and their status lives on the [`theme:m7-observability` label view](https://github.com/kstrat2001/darkmux/issues?q=is%3Aopen+label%3A%22theme%3Am7-observability%22); this page is the reasoning behind it.

[M4](./M4.md)/[M5](./M5.md) make the work trustworthy and [M6](./M6.md) makes the fleet act as one. M7 is where you **see** it: one viewer over the unified stream, fed by telemetry that's actually correct, surfacing anomalies and the value the fleet is producing.

---

## The goal

darkmux already emits a single typed flow stream and ships a viewer that reads it (the fleet + missions lenses, the savings hero, a worktree what-changed summary + editor handoff, presence). M7 completes the observability surface along three lines:

1. **Correct telemetry.** A viewer over wrong data is worse than no viewer. Fix the records that mislead: CPU that measures the container shell instead of inference, a `TeeSink` write that can land in the file but not Redis, live-state derived from the wrong signal.
2. **One drill-down viewer.** Collapse the by-accretion split (the old `topology` / `flow` / `lab` pages plus the retired per-run instrument format) into one viewer that drills fleet → machine → subsystem over the single stream, with the crew roster and mission burn-down the operator actually wants to see.
3. **Surface what matters.** Anomalies (the eureka detectors, transported to the viewer), the token ledger (value made visible), and an alignment spike toward the emerging OpenTelemetry GenAI standard so darkmux interops with the wider observability ecosystem.

The operator's insight from the 1.0 arc is the ordering principle: **the data is only as clean as the loop that produces it.** So correctness leads; visualization follows.

## Why this theme, why now

### What we measured (darkmux's own evidence)

- **Some telemetry measures the wrong thing.** The session view's "process cpu" reads the *container shell*, which is ~0% for LLM work: the inference burns in the host's LMStudio, not the sandbox (#814). A metric that's always zero is worse than absent. It reads as "nothing happening" when everything is.
- **The stream can silently undercount.** `TeeSink` writes each leg independently; a record can land in the local file but fail to reach Redis, so the fleet view under-reports (#798). A stream you observe through is only as trustworthy as it is complete.
- **Live-state derives from the wrong signal.** The viewer's session-level "in-flight" reads a `dispatch.start` with no terminal record, so a hard-killed dispatch lingers as "active." The correct source is presence (`LIVE_SESSIONS`), which the machine view already uses; the session view's inconsistency is the durable fix #857 names.
- **Value is worth making visible, carefully.** The savings hero (tokens a dispatch ran off the metered frontier) shipped and verified live on the real fleet. M7 extends it to a per-loop/mission **token ledger**: **tokens only, never currency** (claiming to save another person money is a liability darkmux doesn't take on; the operator multiplies by their own rate).

### What the research says

- **OpenTelemetry GenAI semantic conventions** are the emerging standard for agent observability: agent spans (create/invoke agent, execute tool, invoke workflow) plus evaluation events. But they're **still in "Development" status** ([OpenTelemetry GenAI semconv](https://opentelemetry.io/docs/specs/semconv/gen-ai/)). The move is therefore *align and map, don't block on stable*: a research spike (#851) maps `FlowRecord` ↔ OTel GenAI so darkmux can interop with the ecosystem without coupling its hot path to an unstable spec, respecting the "payload-not-fields" hold-the-line and the KISS-for-local-AI posture.
- **The observability ecosystem converges** on the same shape darkmux is building: distributed tracing + online evaluation + anomaly detection + session replay. Multi-agent runs emit enough spans per request that raw flat logs stop being practical, which is exactly why drill-down + anomaly surfacing is the right architecture. darkmux's flow stream + eureka detectors are the local-first instance of that convergence.

## Scope

The clusters below explain *why each issue is here*. For current status and the authoritative set, see the [`theme:m7-observability` issues](https://github.com/kstrat2001/darkmux/issues?q=is%3Aopen+label%3A%22theme%3Am7-observability%22).

### Telemetry correctness (leads — the data must be right first)

- **[#814](https://github.com/kstrat2001/darkmux/issues/814) — "process cpu" measures the container, not inference.** Relabel + add a host-side LMStudio CPU series (the scoped fix is a container-stats sampler).
- **[#798](https://github.com/kstrat2001/darkmux/issues/798) — `TeeSink` write isn't atomic.** A record can hit the local file but not Redis; retry / reconcile / surface-failures so the fleet view doesn't undercount.
- **[#857](https://github.com/kstrat2001/darkmux/issues/857) — live-mode from presence, not flow records.** Unify the session view's "in-flight" onto `LIVE_SESSIONS` so a hard-killed dispatch doesn't linger as active.
- **[#813](https://github.com/kstrat2001/darkmux/issues/813) — context chart needs the compaction-threshold marker + y-axis reference** (requires the runtime to emit `threshold` in context telemetry).

## Success criteria

M7 is "done enough" when:

- [ ] No telemetry surface **measures the wrong thing**: CPU reflects inference, the stream doesn't undercount, live-state comes from presence (#814, #798, #857).
- [ ] **One viewer** drills fleet → machine → subsystem over the single stream, showing the crew roster and mission burn-down (#558, #567, #568).
- [ ] **Anomalies surface in the live viewer**, not just at `doctor` time (#657).
- [ ] The **token ledger** attributes off-meter tokens per loop/mission, tokens only, never currency (#732).
- [ ] `FlowRecord` has a **documented mapping to OTel GenAI** conventions, without coupling the hot path to the unstable spec (#851).

Every surface stays **operator-sovereign**: the operator sees the real numbers with provenance, value is shown in tokens (the operator supplies the rate), and anomalies are surfaced for judgment, not auto-actioned ([#44](https://github.com/kstrat2001/darkmux/issues/44)).

## Target release

Per the roadmap principle, [milestones are themes, not version numbers](../../ROADMAP.md) — and the themes are **concurrent lanes, not a sequence**. This theme's work lands **incrementally across the 1.x line**, bundled into whatever release is in flight alongside the other lanes, exactly as 1.0–1.4 already shipped work from every theme at once. No release "owns" a theme. The only hard ordering is the dependency edges named in [Sequence](#sequence); everything else is whatever's ripe.
