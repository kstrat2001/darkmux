PR review: the local review funnel

A local review pipeline that splits the reviewing job across several small dispatches instead of asking one model "any issues?" and hoping. Several probe models each investigate small code bundles under a strong prior, a judge model rules on every flag with a double-confirmation pass, and only what survives becomes a merge-blocking comment — all on local models, no cloud API.

Why a funnel, not a single reviewer

A single local reviewer model asked an open-ended "review this diff" question tends toward shallow output: it restates what the diff obviously does, or flags the bug a fix is removing, or says nothing is wrong because nothing obviously jumped out in one pass. The review funnel restructures the same underlying capability into a pipeline that plays to what small dispatches are actually good at: bounded investigation over a strong prior, not open-ended judgment over an entire diff.

The shape:

  1. Bundle the diff into small, focused code slices — one changed function plus its resolved callees, siblings, and a mechanically-extracted fact sheet (param flow, differentials, sibling comparisons).
  2. Probe each bundle with one or more models, each drawing k independent investigations under the prior "this bundle has been established to contain a defect — find it." A strong prior turns an open-ended search into a bounded one.
  3. Dedup the raw flags: independent draws that land on the same defect collapse into one flag before anyone judges it.
  4. Judge, twice. Every deduped flag gets a first ruling: confirmed, needs a closer look, or false positive. Only a first-pass "confirmed" gets a second, independent judge call over the identical prompt — agreement promotes the flag; disagreement demotes it. A flag never ships as a defect report on the strength of one judge's opinion.
  5. Synthesize the three tiers into a review: confirmed findings become inline, merge-blocking comments; uncertain ones land in a non-blocking "worth a double check" note; everything else is archived in the run's own artifact, never surfaced.
Confirmed still isn't the final word.

Every confirmed finding the funnel posts carries a fixed marker: needs frontier verification — confirmed by a local judge, not yet frontier-verified. A local judge's double-confirm is real signal — cross-context re-thinking, at judge scale — but it isn't the same as a frontier orchestrator's own review. The funnel narrows what the frontier has to look at; it doesn't replace the frontier's judgment on anything that matters.

The funnel never reports a silent "looks fine." A run that confirms nothing still says how much it investigated (how many flags, how many bundles, which models ran it) so a clean result reads as "checked and found nothing," not "didn't check." And a run that produces no usable signal at all — a dead dispatch, an empty reply, a docket that never got off the ground — is marked degraded, never posted as a quiet pass.

Crews: who staffs the funnel

A crew is a saved seat assignment in ~/.darkmux/profiles.json: a top-level sibling of profiles, keyed by crew name, naming which profile (and optionally which explicit model within it) staffs each seat the pipeline needs. The review funnel needs exactly two seat kinds:

  • review-probe: one or more staffings, each drawing k independent investigations per bundle. A staffing can optionally carry a bundle_selector that scopes it to specific fact families (e.g. only security/correctness bundles) instead of the full set — useful for pairing a wide, cheap probe against everything with a narrower, more expensive one against the bundles that matter most.
  • review-judge: exactly one staffing. It rules on every deduped flag, twice for anything it confirms on the first pass.

The shipped example crew, from profiles.example.json:

{
  "profiles": {
    "balanced": {
      "description": "Mid-range tasks. Predictable behavior across mixed workloads.",
      "models": [{ "id": "<your-worker-model-id>", "n_ctx": 100000 }]
    },
    "deep": {
      "description": "Long agentic tasks. Maximum worker context for fewer compactions.",
      "models": [{ "id": "<your-worker-model-id>", "n_ctx": 200000 }]
    }
  },

  "crews": {
    "review-deep": {
      "description": "Deep review lineup: two probe staffings (one plain, one bundle-scoped) plus a single judge staffing.",
      "seats": {
        "review-probe": [
          { "profile": "balanced" },
          { "profile": "deep", "k": 1, "bundle_selector": { "fact_families": ["security", "correctness"] } }
        ],
        "review-judge": [
          { "profile": "deep" }
        ]
      }
    }
  }
}

Here balanced staffs a wide instruct probe (draws over every bundle, default k) and deep staffs a second, bundle-scoped probe (a single deep-reasoning draw restricted to the security/correctness fact families) — plus the same deep profile staffs the judge seat. Replace the model ids with what you have downloaded (lms ls). The self-review runner doc has a fuller worked example naming real model ids for a three-model, 32GB-tier crew.

The seat law

The judge seat takes the call volume: every deduped flag gets a pass-1 judge call, and every confirmed pass-1 gets a second, independent pass-2 call. On a busy diff that can be a lot of judge dispatches from a small number of probe flags. Staff the judge seat with a fast, high-throughput model — an MoE-class model with a small active-parameter count reads well here, since it needs to rule quickly and repeatedly, not reason exhaustively over a whole diff. Slow, deep reasoners belong in probe seats, where a bundle_selector keeps their few, expensive draws scoped to the fact families that benefit most from careful reasoning, instead of running them over every bundle in the diff.

Inspect a crew

There's no dedicated inspection verb for the crews map yet (darkmux crew list is a different thing — it lists roles, the crew-dispatch abstraction covered in missions, sprints, and crew dispatch). Read the registry directly:

jq '.crews' ~/.darkmux/profiles.json

Run it locally

darkmux pr-review run drives the funnel end-to-end: bundle the diff, dispatch the probe/judge seats, and emit the same {mode, review, comment} payload the older single-reviewer pr-review render path produces.

darkmux pr-review run \
  --worktree /path/to/local/checkout \
  --diff pr.diff \
  --crew review-deep \
  --mode auto \
  --envelope-out funnel.json \
  --emit rendered.json

Source the bundler's file content from either a local checkout (--worktree <dir>) or the GitHub API with no checkout at all (--github <owner/repo> --head-sha <sha>) — the two are mutually exclusive. --intent-file optionally feeds the diff author's stated intent (title + description) into every probe/judge prompt, so the funnel assesses the diff against its stated purpose instead of flagging the bug a fix is removing.

What comes back

The rendered payload's mode field is one of three:

  • review: at least one confirmed finding. Posted as REQUEST_CHANGES with inline comments anchored to diff lines (or a general body item when a finding's quoted anchor can't be resolved to exactly one line — never a guessed line number). Each confirmed comment carries the needs frontier verification marker. Any needs_check flags ride along as a non-blocking "worth a double check" section in the same body.
  • comment: zero confirmed findings on a healthy run. An honest summary — how many flags were investigated, across how many bundles, by which models — plus any needs_check flags as non-blocking notes. Never a silent pass with no signal attached.
  • degraded: the run produced no usable signal at all (a dead dispatch, an empty reply, a docket that never got off the ground). Posted as a loud warning, never a green check.

The envelope artifact

--envelope-out <path> writes the full run record: every probe flag, every judge ruling (pass-1 and, where it ran, pass-2), the tier each flag landed in, per-model wall-clock and token telemetry, and a fingerprint (judge model + temperature + protocol version) two envelopes need to share before their tiers are comparable. It's the audit trail behind the rendered comment — keep it if you want to inspect what got archived, or replay synthesis later without redispatching (see --from-envelope below).

Exec modes: sequential, parallel, auto

The --mode flag governs LMStudio residency across the funnel's dispatches, not concurrent network calls (dispatches within a mode still run one at a time):

  • sequential: loads one seat's model, runs every draw for it, releases it, then moves to the next. Fits machines that can't hold every distinct model in the crew resident at once.
  • parallel: loads every distinct model in the crew up front and keeps them all resident for the whole run. Needs enough RAM headroom to hold the crew's full model set simultaneously.
  • auto (default): resolves to one of the above, once, up front, from the machine's detected hardware tier and how many distinct models the crew needs resident (a probe and the judge sharing the same profile/model count once, not twice). The resolved choice is recorded in the envelope's mode field, so you never have to wonder which one actually ran.

--k overrides every review-probe staffing's draw count for this run, without editing the crew. --bundler <cmd> swaps in an external bundler (any executable implementing <cmd> --worktree <dir> --diff <file> and emitting the same frozen bundle JSON) in place of the built-in one. --from-envelope <path> re-runs synthesis only against a previously written envelope — zero model calls, zero bundling — the escape hatch for re-rendering a payload without redispatching.

CI recipe

darkmux reviews its own PRs this way: .github/workflows/darkmux-review.yml on a self-hosted runner, dogfooding the funnel in public. The pattern generalizes to any repo with a self-hosted runner and a local model stack.

gh workflow run darkmux-review.yml -f pr=<PR_NUMBER>
# optional overrides:
#   -f crew=<name>   which crew to dispatch (default: a repo variable, falling back to "review-deep")
#   -f mode=<mode>   sequential | parallel | auto
#   -f k=<n>         override draws-per-seat for every staffing in the crew

Three things worth knowing about how the workflow is built:

  • Inputs are pr, crew, mode, k. The crew — and therefore which models actually run the review — lives entirely in the runner's own ~/.darkmux/profiles.json, never in the workflow file. The public YAML never names a model.
  • No PR checkout. The job reads only the PR's diff, title, and description via the GitHub API — data, never executed — and passes --github <owner/repo> --head-sha <sha> so the funnel's own file source fetches any additional file content it needs the same way. The only checkout is of the repo's trusted default branch (the review tooling itself), never the reviewed PR's code.
  • The envelope uploads as an artifact. Whatever --envelope-out wrote goes up via actions/upload-artifact regardless of outcome (if: always()), so a degraded or oddly-scored run is still inspectable after the fact.

Trigger is deliberately workflow_dispatch-only: it never fires on a bare pull_request event, so a stranger's PR (or a fork) can't launch anything on the runner. Only someone with write access runs it. See DARKMUX_REVIEW_RUNNER.md for the full runner setup (registering the self-hosted runner, prerequisites, and a copy-pasteable crew migration example).

Validate your crew before trusting it

A crew is a claim about model behavior: "these probe models plus this judge produce trustworthy confirmed findings." Don't take that on faith. darkmux lab review-bench --funnel runs the same funnel machinery over a labeled case corpus and scores it — precision, recall, verdict, and anchor accuracy — exactly the way every other review mode in the bench is scored.

darkmux lab review-bench \
  --funnel \
  --crew review-deep \
  --workdirs /path/to/case-workdirs \
  --cases-dir templates/builtin/lab-fixtures/pr-review-bench/cases

--workdirs points at an evidence root with one subdirectory per case id holding that case's repository tree at the reviewed commit (the probe/judge seats read real code, the same way they would against a live PR). --exec-mode (the bench's name for pr-review run's --mode) and --k let you bench the exact exec-mode/draw-count combination you intend to run in CI. Run this whenever you change a crew's staffing — a new probe model, a different judge, a changed bundle_selector — before pointing production PRs at it.

Hardware sizing

The funnel's RAM footprint under parallel is the sum of every distinct model the crew needs resident; under sequential it's just the largest single model, since only one is ever loaded at a time. Distinct means distinct model id — two staffings pointing at the same profile's default model count once, not twice. Two illustrative examples for a crew staffing three genuinely distinct models (two differently-modeled probe staffings plus a separately-modeled judge, the shape the self-review runner's example crew uses):

MachineWhat auto resolves toWhat happens
32 GB sequential One probe model loads (say, roughly 14 GB for a dense model in the 20-30B class at 4-bit quantization), runs its draws, releases. The second probe staffing's model loads, runs, releases. The judge's model loads last, rules on every deduped flag (twice for anything it confirms), releases. Never more than one model resident.
128 GB parallel All three distinct models — both probe staffings' models and the judge's — load up front and stay resident for the whole run. No load/release churn between seats; dispatches still run one at a time, but there's no reload cost between them.

auto makes this call from a coarse RAM-tier budget against the crew's distinct-model count: machines up to 32 GB get a budget of one resident model, the 33-64 GB tier gets two, and above 64 GB gets three. A crew needing more distinct models than the budget allows falls back to sequential regardless of tier. Pass --mode sequential explicitly if you want the conservative, one-at-a-time behavior on a machine that could technically fit more — useful when the funnel is sharing the box with something else that also wants RAM.
