Getting started

Install darkmux, verify the setup, configure your first profile, swap to it, and run a smoke test. Everything else in the guide assumes you've done this.

Prerequisites

RequiredWhyInstall
LMStudio darkmux drives LMStudio via the lms CLI for load/unload. macOS / Windows / Linux installer.
At least one model in LMStudio Nothing to swap to without one. Download via the LMStudio UI; verify with lms ls.
Rust toolchain To build darkmux itself. curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh

darkmux is developed and tested on Apple Silicon. Linux should work; Intel Mac is untested.

Optional (but unlocks a lot)

ToolWhen you need it
An agent runtime (e.g. OpenClaw) For darkmux lab dispatches and darkmux crew dispatch. Pure swap/status/profiles work without one. Override the default with DARKMUX_RUNTIME_CMD=<your-runtime>.
Claude Code (or any frontier orchestrator) For agent-invokable skills and dispatching workflows. darkmux works as a CLI without it.

Install + bootstrap

One copy-pasteable block

Works from a fresh machine that has LMStudio installed:

# 1. Install Rust toolchain (skip if `cargo --version` already works)
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
source "$HOME/.cargo/env"

# 2. Clone + build darkmux
git clone https://github.com/kstrat2001/darkmux
cd darkmux
cargo install --path .

# 3. Bootstrap config + agent skills
darkmux init
Heads up.

The source "$HOME/.cargo/env" line is the one most often missed. Without it, a fresh cargo install fails with command not found: cargo in the same shell that just ran the rustup installer.

Verify your setup

darkmux doctor runs pre-flight checks: profile registry, lms binary, models loaded, runtime command, RAM headroom, power state, agent role scaffolds, daemon reachability, and a battery of empirical rules from the eureka set.

darkmux doctor

Each check returns Pass, Warn, or Fail. Fail-level lines include actionable hints; the exit code is 0 on all-pass-or-warn, 1 if anything Fails.

Output is one line per check, color-coded: ✓ for Pass, ! for Warn, ✗ for Fail. Run it yourself on a fresh setup — that's the canonical reference for what it looks like.

Common first-run warnings:

  • daemon: reachable Warn — you haven't started darkmux serve yet. That's fine for now; you'll start it in the Observability section.
  • profile match Warn — your ~/.darkmux/profiles.json doesn't yet reference what's loaded in LMStudio. That's also fine — we fix it next.
  • runtime version Warn — your installed agent runtime is older than darkmux's validated minimum. If you're not running an orchestrator, you can ignore this for now.

Configure your first profile

A profile is a named loadout — which model(s) to keep resident in LMStudio at which context length. Profiles live in ~/.darkmux/profiles.json (created by darkmux init).

Inspect what's already there

darkmux profiles                  # list configured profiles
cat ~/.darkmux/profiles.json      # see the actual JSON

The bootstrap config ships with placeholder profiles (fast, balanced, deep, plus a gpt-oss preset for the 120B MXFP4 build on Apple Silicon). Each profile lists models with placeholder IDs like <your-primary-model-id>.

Swap a real model id in

Run lms ls to see what's downloaded in LMStudio. Pick one (start with whatever you've been using day-to-day) and replace a placeholder in profiles.json:

# See what's available
lms ls

# Edit the file (use whatever editor)
${EDITOR:-vim} ~/.darkmux/profiles.json

Each profile model has three fields that matter:

  • id — the LMStudio model id from lms ls output (e.g. qwen3.6-35b-a3b).
  • n_ctx — context length to load at. Defaults are usually fine; 32768 is a safe starter for any model.
  • role — one of primary, compactor, auxiliary. Every profile needs exactly one primary; the rest are optional.

Verify the edits

darkmux profiles                  # the profile should now list your model
darkmux doctor                    # confirm "profile match" is green or close

Your first swap

darkmux swap <profile> atomically transitions LMStudio's loaded model set to match the named profile. It unloads anything that doesn't belong, loads what does, and patches your agent runtime's config so the agent requests the right model identifier on its next dispatch.

darkmux status                    # what's loaded right now; which profile (if any) matches
darkmux swap fast                 # load the "fast" profile
darkmux status                    # confirm — should show "fast" as the active profile
What just happened.

darkmux issued lms load for your profile's primary model with a darkmux: namespace prefix (so darkmux can recognize its own loads later), set the context length, updated ~/.openclaw/openclaw.json (or your runtime's config) to reference the now-loaded identifier, and reported success. Your orchestrator's next dispatch will route to this model.

If swap reports a warning about RAM headroom or a model not found in lms ls, that's doctor-style feedback at swap time — usually fixable by either trimming context length or downloading the model first via the LMStudio UI.

The smoke test

darkmux lab run quick-q           # single-turn smoke prompt against the active profile
darkmux lab inspect <run-id>      # quick look at what happened

This requires an agent runtime (OpenClaw or override). If you don't have one configured, skip this step — you can come back to it after reading the lab section.
